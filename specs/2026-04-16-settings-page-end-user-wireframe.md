# Wireframe: страница настроек для конечных пользователей

## 0. Метаданные
- Тип (профиль): `dotnet-desktop-client`
- Overlay profile: `product-system-design`
- Владелец: Unlimotion desktop settings UX
- Масштаб: `medium`
- Статус: `approved for EXEC`
- Цель артефакта:
  - согласовать новую структуру страницы настроек до XAML-реализации;
  - убрать developer-centric UX;
  - сделать экран понятным обычному пользователю с первого открытия.
- Ограничения:
  - на этом шаге не меняем код; документ может требовать будущих изменений binding/commands в реализации;
  - wireframe проектируется для текущей desktop-вкладки `Settings`;
  - технические параметры Git допускается скрыть или увести в `Advanced`, если они не нужны основной аудитории.
- Связанные файлы:
  - `src/Unlimotion/Views/SettingsControl.axaml`
  - `src/Unlimotion.ViewModel/SettingsViewModel.cs`
  - `src/Unlimotion.ViewModel/AppearanceSettings.cs`
  - `src/Unlimotion/App.axaml.cs`
  - `src/Unlimotion/Views/MainControl.axaml`

## 1. Overview / Цель
Сделать страницу настроек настолько очевидной, чтобы пользователь без инструкции понимал:
- где хранятся его задачи;
- включена ли синхронизация и backup;
- что безопасно менять самому;
- какие действия редкие, опасные или нужны только для поддержки.

## 2. Текущее состояние (AS-IS)
- Все настройки находятся в одном длинном вертикальном списке.
- Локальное хранилище, сервер, Git backup и SSH смешаны без явных секций.
- Пользователь видит технические сущности раньше пользовательских задач.
- Опасные действия визуально не отделены от обычных.
- Неясно, что сохраняется автоматически, а что начинает работать только после `Connect`.
- Названия полей и кнопок ориентированы скорее на разработчика, чем на конечного пользователя.

## 3. UX-принципы целевого экрана
- Сначала пользовательский смысл, потом технические детали.
- Один вопрос = один блок.
- Видны только релевантные поля для текущего режима.
- У каждой секции есть статус: что сейчас включено и работает ли это.
- У каждой секции есть понятный следующий шаг.
- Продвинутые параметры скрыты по умолчанию.
- Опасные действия убраны в отдельный блок с предупреждением.
- Язык интерфейса простой, без лишнего Git- и infrastructure-жаргона.

## 4. Информационная архитектура (TO-BE)
### 4.1 Основные секции
1. `Внешний вид`
2. `Хранилище задач`
3. `Резервное копирование`
4. `Расширенные настройки`
5. `Обслуживание и восстановление`

### 4.2 Что должен видеть обычный пользователь
- Тема
- Размер текста
- Где хранятся задачи: `На этом устройстве` или `На сервере`
- Состояние подключения
- Кнопку `Выйти`, если пользователь уже подключен к серверу
- Включен ли backup
- Способ входа в backup: `Токен` или `SSH`
- Кнопка ручной синхронизации

### 4.3 Что должно быть скрыто по умолчанию
- `Git Remote Name`
- `PushRefSpec`
- `Committer Name`
- `Committer Email`
- SSH-список ключей, если не выбран SSH-способ входа
- `Migrate`, `Resave` и другие сервисные операции

### 4.4 Явные изменения модели состояния
- Тема интерфейса перестает быть бинарной настройкой.
- Вместо текущего `bool IsDarkTheme` целевая модель должна поддерживать 3 состояния:
  - `System`
  - `Light`
  - `Dark`
- Способ входа в backup должен быть выражен как явное UI-состояние:
  - `Token`
  - `Ssh`
  - состояние вычисляется автоматически;
  - если выбран remote из локального repo, тип определяется по URL этого remote;
  - если repo еще не создан или remote недоступен, тип определяется по `GitRemoteUrl`.
- Серверный блок должен поддерживать явные состояния:
  - `Не подключено`
  - `Подключение...`
  - `Подключено как <логин>`
  - `Ошибка подключения`
- Backup-блок должен поддерживать явные состояния:
  - `Не настроено`
  - `Подключение...`
  - `Подключено`
  - `Синхронизация...`
  - `Ошибка синхронизации`

## 6. Поведение секций

### 6.1 Внешний вид
- `Dark theme` заменить на `Тема интерфейса`.
- Вместо бинарного чекбокса использовать `ComboBox`: `Как в системе / Светлая / Темная`.
- Это требует замены текущей темы-модели с `bool` на tri-state / enum.
- `Font Size` оставить быстрым и наглядным, с шагами `- / +` и небольшим preview.

### 6.2 Хранилище задач
- `Server Mode` убрать как технический чекбокс.
- Вместо него использовать выбор режима в человеческой формулировке:
  - `На этом устройстве`
  - `На сервере`
- Показывать только поля активного режима.
- Кнопка `Connect` должна называться `Подключить`.
- Под кнопкой нужен явный статус:
  - `Подключено`
  - `Не подключено`
  - `Ошибка подключения`
- Если уже есть активный серверный сеанс, показывать:
  - `Подключено как <логин>`
  - кнопку `Переподключить`
  - кнопку `Выйти`

### 6.3 Резервное копирование
- `Git Backup Enabled` заменить на пользовательский смысл: `Автоматически создавать резервную копию`.
- Backup настраивается на одной странице, без отдельного мастера.
- В блоке должен быть field `Способ входа`, который определяется автоматически.
- Если в локальном repo доступно несколько remotes, рядом должен быть selector `Удаленный источник`, и смена remote должна автоматически менять auth type.
- Для входа по токену базовый сценарий должен умещаться в поля:
  - адрес репозитория;
  - ветка;
  - логин;
  - токен доступа;
  - кнопка подключения.
- Для входа по SSH базовый сценарий должен умещаться в поля:
  - адрес репозитория;
  - ветка;
  - автоматически определенный способ входа;
  - блок `SSH-ключи`;
  - кнопка подключения.
- `Clone`, `Pull`, `Push` не стоит показывать как три равноправные кнопки по умолчанию.
- Для пользователя лучше:
  - `Подключить репозиторий`
  - `Синхронизировать сейчас`
- Автоматические интервалы и служебные Git-параметры увести в `Дополнительные параметры Git`.
- Термин `Пароль` для backup не использовать, если фактически ожидается токен.
- У backup-секции должны быть состояния:
  - `Не настроено`
  - `Подключение...`
  - `Подключено`
  - `Синхронизация...`
  - `Ошибка синхронизации`

### 6.4 SSH
- Показывать только при выбранном SSH-способе входа.
- Не использовать формулировку `Existing SSH public keys`.
- Простая подача:
  - `Активный ключ`
  - `Создать новый ключ`
  - `Копировать публичный ключ`

### 6.5 Обслуживание
- `Migrate`, `Backup`, `Resave` нельзя держать рядом с обычными настройками.
- Этот блок должен быть отдельным, свернутым и с предупреждением.
- Если later implementation позволит, для сервисных действий нужен confirm-dialog.
- Названия действий должны совпадать с их реальным эффектом, а не быть абстрактными.

### 6.6 Матрица сохранения и применения
| Настройка | Сохраняется | Применяется |
| --- | --- | --- |
| Тема интерфейса | Сразу | Сразу |
| Размер текста | Сразу | Сразу |
| Умный поиск | Сразу | Сразу |
| Режим хранения задач | Сразу | После `Подключить` |
| Папка с данными | Сразу | После `Подключить` |
| Адрес сервера / логин / пароль | Сразу | После `Подключить` |
| Параметры backup | Сразу | После `Подключить репозиторий` |
| Сервисные действия | Не автоприменяются | Только по явной кнопке и confirm |

### 16.5 Решение по `BackupAuthMode`
- `BackupAuthMode` не добавляется как persisted поле в `GitSettings`.
- `BackupAuthMode` остается runtime derived state.
- Источник вычисления:

  - смена remote автоматически меняет отображаемый auth type;
