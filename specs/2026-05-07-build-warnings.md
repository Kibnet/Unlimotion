# Уменьшение простых предупреждений сборки

## 0. Метаданные
- Тип (профиль): delivery-task; profiles: `dotnet-desktop-client`, локальный cleanup без изменения поведения
- Владелец: Codex
- Масштаб: small / medium
- Целевая модель: gpt-5.5
- Целевой релиз / ветка: текущая рабочая ветка
- Ограничения: минимальные точечные изменения; не менять публичный API; не исправлять предупреждения, требующие крупной переработки архитектуры, UI-потоков или продуктовых решений
- Связанные ссылки: `AGENTS.md`, `C:\Projects\My\Agents\AGENTS.md`, `AGENTS.override.md`

Если секция не применима, явно укажите `Не применимо` и короткую причину, вместо заполнения нерелевантными деталями.

## 1. Overview / Цель
Уменьшить максимальное количество предупреждений сборки, которые можно безопасно убрать локальными правками без большого объёма изменений.

Outcome contract:
- Success means: `dotnet build` показывает меньше предупреждений, чем baseline, а внесённые изменения не меняют поведение приложения.
- Итоговый артефакт / output: точечные правки кода/проектных настроек и отчёт со списком проверок.
- Stop rules: остановиться, если оставшиеся предупреждения требуют архитектурных изменений, обновления зависимостей с высоким риском, изменения публичного API или длительного UI/product investigation.

## 2. Текущее состояние (AS-IS)
- Решение находится в `src/Unlimotion.sln`.
- Проект использует .NET, Avalonia, TUnit и централизованные версии пакетов через `src/Directory.Packages.props`.
- Конкретный набор предупреждений должен быть снят фактическим baseline-запуском `dotnet build`.
- Локальный `AGENTS.override.md` требует UI-тесты для UI-facing изменений. Планируемые правки не должны менять UI-поведение; если это изменится, нужно добавить/обновить UI-тесты.

## 3. Проблема
Сборка содержит предупреждения, часть которых можно убрать малыми локальными изменениями, чтобы снизить шум и облегчить поиск новых регрессий.

## 4. Цели дизайна
- Разделение ответственности: исправлять предупреждения рядом с их источником.
- Повторное использование: не вводить новые абстракции без необходимости.
- Тестируемость: проверять результат сборкой и релевантными тестами.
- Консистентность: сохранять стиль существующего кода и централизованные настройки проекта.
- Обратная совместимость: не менять публичные контракты и пользовательское поведение.

## 5. Non-Goals (чего НЕ делаем)
- Не выполнять крупные архитектурные refactor.
- Не менять UI-потоки, визуальное поведение или automation selectors.
- Не обновлять крупные зависимости только ради подавления предупреждений.
- Не переводить предупреждения в глобальные suppressions, если можно сделать простую безопасную правку.
- Не исправлять platform-specific warning, если для проверки нужен отдельный SDK/workload, отсутствующий локально.

## 6. Предлагаемое решение (TO-BE)
### 6.1 Распределение ответственности
- Файлы с предупреждениями -> точечные локальные правки.
- Проектные настройки -> только если предупреждение связано с конфигурацией сборки и правка не маскирует проблему.
- Spec -> журнал решений, quality gate и результат проверок.

### 6.2 Детальный дизайн
- Снять baseline через `dotnet build src/Unlimotion.sln`.
- Сгруппировать предупреждения по коду и риску.
- Исправить низкорисковые предупреждения: nullable, неиспользуемые using/переменные, naming/style, очевидные analyzer suggestions, дублирующиеся package references, устаревшие API только при локальной замене без изменения поведения.
- Повторить сборку и сравнить количество предупреждений.
- Запустить релевантные тесты; полный прогон выполнить, если он локально доступен за разумное время.

## 11. Тестирование и критерии приёмки
- Acceptance Criteria:
  - Baseline warning count зафиксирован.
  - После правок warning count уменьшился.
  - `dotnet build src/Unlimotion.sln` завершился успешно или оставшиеся ошибки/ограничения явно описаны.
  - Релевантные тесты запущены; если UI-поведение не менялось, новые UI-тесты не требуются.
- Какие тесты добавить/изменить: не планируется, потому что задача должна сохранять поведение. Добавить тест только если правка затрагивает поведение.
- Characterization tests / contract checks: существующие тесты и сборка.
- Базовые замеры до/после для performance tradeoff: не применимо.
- Команды для проверки:
  - `dotnet build src/Unlimotion.sln`
  - релевантный `dotnet test`/`dotnet run --project <test project>` после понимания runner workflow
- Stop rules для test/retrieval/tool/validation loops: не гонять бесконечно полный suite, если он зависает или требует внешнего окружения; в отчёте указать next-best check.

## 12. Риски и edge cases
- Устаревшие API могут иметь неочевидную замену; такие предупреждения пропускать, если замена не локальна.
- Android/iOS предупреждения могут зависеть от установленного workload.
- Nullable-правки могут скрыть реальную проблему, если использовать `!` без доказанного инварианта; предпочитать проверку или локальную инициализацию.

## 13. План выполнения

3. Внести точечные правки.
4. Повторить сборку и тесты.
5. Выполнить post-EXEC review и обновить журнал.
